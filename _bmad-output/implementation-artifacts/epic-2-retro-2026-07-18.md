## What Went Well

1. **"Previous Story Intelligence" habit held.** All 3 Epic 2 stories carried this section forward, continuing the practice committed to in the Epic 1 retro.
2. **Layer-boundary discipline caught early and stuck.** Story 2.1's review caught a Presentation-layer bypassing server actions and a TOCTOU race in ownership-scoped updates; both were fixed immediately and no later story repeated them.
3. **Good architectural catch in Story 2.2.** The dev agent discovered a genuine routing conflict (a plain `src/app/page.tsx` silently shadowing `(student)/page.tsx`) and resolved it cleanly by extracting a shared `resolveActiveChildProfile()` helper — prevented a real dead-code trap from persisting.
4. **Security gap found and closed proactively.** Story 2.2's review noticed the `child-profile-id` cookie was never cleared on sign-out (a cross-parent data exposure risk on shared devices) — and since no sign-out flow existed at all, the team built one on the spot rather than leaving the gap.

## Challenges / Growth Areas

1. **No live browser verification across all 3 stories.** Every story's Dev Agent Record documents the same root cause: no seeded PARENT/ChildProfile fixtures and no DB connectivity in the sandbox, forcing verification down to `next build` + `curl` + token-level CSS checks. This retrospective's own live walkthrough with Toan surfaced two real, previously-unseen bugs that had shipped as "done":
   - A Grade Band dropdown that silently failed to show its placeholder, because the form's state (`GradeBand | ''`) didn't match Base UI Select's `null`-only "no selection" contract (Base UI's Select API is meaningfully different from the older Radix-based shadcn select this codebase was implicitly modeled on).
   - No navigation from the profile switcher into the student home screen — Story 2.2's AC only required the cookie to be set and the parent header to refresh, so nobody wired the actual "go to student view" step.
   Both were fixed and confirmed working during this retrospective.
2. **Test framework is now blocking, not just a future concern.** The Epic 1 retro flagged this for "before Epic 3 starts" — that time has arrived, and Story 3.1 has a hard AC requiring 10+ unit tests with nowhere to land them.
3. **Stale epic-status flag recurred.** `epic-1`'s flag stayed `in-progress` after all stories were `done` (flagged in the Epic 1 retro); the exact same thing happened again with `epic-2`. No automatic transition exists — this is the second time a human had to notice and fix it manually.
4. **Scope grew organically in Story 2.2** beyond its original task list (sign-out flow, Edge Middleware, signature extraction) — a good catch each time, but a sign that review is finding real gaps the story planning didn't anticipate.
5. **A known visual risk (`rounded-brand-xl` vs. `tailwind-merge`) has now been deferred twice** (Story 2.2's switcher, Story 2.3's home card) without being addressed.

## Epic 3 Readiness

**No significant discoveries require updating Epic 3's plan.** Architecture assumptions (cookie handoff, layer rules) held through Epic 2's build.

**Real readiness gap:** Epic 3 is entirely student-facing interactive UI (mascot reactions, feedback overlays, audio playback, offline resilience). Going in with the same "no live verification" pattern that just produced two shipped-but-broken UI issues in Epic 2 is a materially higher risk for Epic 3's scope.

## Closing

Epic 2 delivered 3/3 stories, with solid architectural discipline (layer rules, ownership scoping, a proactively-caught sign-out security gap). The main lesson: the sandbox's inability to do live browser verification is no longer a theoretical risk — it produced two real, user-facing bugs that shipped as "done" and were only caught by a manual walkthrough during this retrospective. Epic 3's fully interactive student UI raises the stakes on this gap considerably, and the test-framework action item (now critical path, owned directly by Toan) is the first step toward closing it.
