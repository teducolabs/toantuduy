No 30-second decomposition across FR-13–FR-17

Vision §1, §4.4

Fix: Add a consequence to FR-14 or the Parent Dashboard section description: "A parent can identify the weakest Skill for the active Child Profile in ≤ 3 taps / ≤ 30 seconds from Dashboard open."

2. Substance over theater

Adequate

Personas drive decisions: Minh → FR-1/FR-5 edge cases; Lan → FR-13/FR-14/FR-15 specificity; Cô Hương → FR-20/FR-21 workflow. NFRs carry real thresholds (500ms, 2s, 1s, 3s, 60s, 44×44px). High

SM-4 is measurement theater

§8 SM-4
"Child Profiles with ≥ 2 weeks of Sessions show measurable improvement in word problem accuracy" has no baseline protocol, no delta threshold, no comparison group. SM-4 is the only metric validating the adaptive engine (FR-7–FR-9). As written, it passes with any positive accuracy delta, even noise.
Fix: Define a minimum meaningful delta (e.g., "per-Skill accuracy improves by ≥ 10 percentage points from Sessions 1–5 to Sessions 11–15 for Child Profiles with ≥ 15 Sessions") and a baseline condition (first-2-Sessions accuracy vs. Sessions 11–15).

Strong

The thesis is explicit: free Teacher Portal → teacher referral → parent acquisition → student practice → learning signal validates product. Every major feature cluster serves one leg of this arc. Counter-metrics are named and their anti-gaming logic is explained. Success Metrics link back to specific FRs. No findings.

4. Done-ness clarity

Thin

Most FRs are adequately specified. The session history / learning history cluster (FR-7, FR-13–FR-15, FR-17) carries disproportionate ambiguity. FR-15's sole testable consequence is untestable (broken). FR-7's accuracy model has an undefined unit and an unexamined early-anchor effect. FR-14's drill-down meaning is ambiguous between two interpretations that tell completely different stories to a parent.

FR-7 (per-Skill accuracy tracking) — thin: "Running accuracy (correct / total attempts)" — "attempt" is undefined. The all-time cumulative model also silently penalizes children who improve, a deliberate product choice never named as such.

FR-13 (weekly activity summary) — adequate, one gap: UJ-2 shows "streak intact" as visible parent-facing UI, but no FR defines what a streak is or requires it. FR-13 covers count and day-presence, not streak.

FR-14 (Skill breakdown view) — thin: "Per-Session accuracy" in the drill-down is ambiguous between Skill accuracy within that Session (a trend) vs. running accuracy at time of Session (a slowly-moving average). No minimum-attempts floor before a Skill is categorized.

FR-15 (grade progress indicator) — broken: "Normalized within the Grade Band" is untestable. Difficulty Level ordinal scale, normalization function, and label-to-value mapping are all undefined. UX cannot design this; engineering cannot implement it.

FR-17 (Session history) — adequate, one gap: No data retention policy for Sessions beyond the 30-Session display floor on active profiles.

Critical

FR-15 normalization undefined — untestable consequence

§4.4 FR-15
Consequence 2 states "normalized within the Grade Band." Three things are undefined: (1) how many Difficulty Levels exist per Grade Band — the Glossary says "ordinal rating" with no count or names; (2) what "normalized" means — percentile, min-max, lookup table; (3) how a normalized value maps to the label "Đang ở: Lớp 2, giữa kỳ" — this label system (đầu kỳ / giữa kỳ / cuối kỳ) is undefined. A UX designer cannot design this widget. An engineer cannot implement it. A QA tester cannot verify it.
Fix: Define the Difficulty Level ordinal scale in the Glossary (e.g., 1–5 per Grade Band). Rewrite FR-15 consequence 2 with the actual normalization formula and label-to-value mapping. Add a [NOTE FOR PM] if the label system is still TBD.
High

FR-7 "attempt" granularity undefined

§4.2 FR-7
"Running accuracy (correct / total attempts)" — "attempt" is never defined in the Glossary or FR. If a Session has 4 Pattern Recognition Questions and 3 are correct, is that 4 attempts at 75%? This is the arithmetic foundation for FR-8 (difficulty selection) and FR-14 (Tốt/Cần luyện thresholds). Downstream implementers will make incompatible choices.
Fix: Add to FR-7 consequences: "An attempt is defined as a single Question answered within a completed Session. Questions in incomplete Sessions are not counted."
High

FR-7 cumulative accuracy model unnamed — early-anchor effect

§4.2 FR-7
"Accuracy reflects all historical attempts" — a child who answered their first 5 Pattern Recognition Questions at 20% correct must answer 45 more at 100% to reach the 70% "Tốt" threshold. A parent will see "Cần luyện" for a Skill the child has mastered in recent weeks. This is a deliberate product choice that has never been named as such, directly affecting parent trust (SM-2).
Fix: Add a [NOTE FOR PM]: "All-time cumulative accuracy means a child who improves significantly will continue to see 'Cần luyện' for many Sessions. Confirm this is the intended model vs. a recency-weighted approach (e.g., last-N-Sessions accuracy)."
High

FR-14 "per-Session accuracy" ambiguous — trend vs. snapshot

§4.4 FR-14
"Tapping a Skill shows the last 3 Sessions… with per-Session accuracy." Two interpretations: (a) accuracy for that Skill within that Session specifically (e.g., "2/3 correct") — a trend; (b) running accuracy at time of that Session — a slowly-moving average. A parent judging "is my child improving?" needs (a). These tell completely different stories and must be disambiguated before UX design.
Fix: Specify: "Tapping a Skill shows the last 3 Sessions that included Questions from that Skill, with the Child Profile's accuracy for that Skill within that Session only (e.g., '2/3 correct')."
Medium

FR-14 no minimum-attempts floor before Skill is categorized

§4.4 FR-14
"All Skills the Child Profile has encountered at least once appear in the breakdown." A child who answered 1 Question and got it right shows 100% "Tốt" — a misleading signal to the parent on Day 1.
Fix: Add a consequence: "A Skill appears in the breakdown only after the Child Profile has attempted ≥ [N] Questions from that Skill. Prior to that threshold, the Skill is listed as 'Chưa đủ dữ liệu' (insufficient data)." Mark [N] as a new [ASSUMPTION: A-12].
Medium

FR-17 no retention policy beyond 30-Session display floor

§4.4 FR-17
Active Child Profile Session history has no stated upper bound. A parent with 18 months of daily Sessions — are Sessions 31+ accessible by scrolling or dropped? The Glossary's Child Profile entry says it "holds Session history" without bounds. Contrast with A-3 (deleted profile retained 30 days) — active profile history has no limit stated.
Fix: Add to FR-17: "All completed Sessions are retained for the lifetime of the Child Profile. The display shows the most recent 30 by default with pagination/scroll to older entries."
Low

FR-4 single-Skill Session summary undefined

§4.1 FR-4
"Summary includes a per-Skill breakdown if the Session contained Questions from more than one Skill." Teacher-assigned sets targeting a single Skill produce Sessions where this condition is false — the single-Skill case is left unspecified.
Fix: Add: "If the Session contained Questions from only one Skill, the Session Summary shows that Skill's accuracy (e.g., '6/8 Pattern Recognition')."
