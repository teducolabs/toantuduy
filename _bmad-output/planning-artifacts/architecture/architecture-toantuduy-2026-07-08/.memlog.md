- (decision) AD-1: Full-stack Next.js monorepo — no separate backend service. Binds all surfaces. Prevents a frontend/backend split that would add a separate deployment, versioned API contract, and auth propagation layer on day one. Rule: all server-side logic (mutations, queries, business rules) lives in the Next.js monorepo via server actions and route handlers.
- (decision) AD-2: Pragmatic clean architecture. Domain logic in src/domain/ (entities, use cases), repositories in src/infrastructure/repositories/ wrapping Prisma, server actions in src/app/**/actions.ts as the application boundary. No formal port/adapter interfaces. Rule: server actions may call domain use cases and repositories; React components may NOT import from src/domain/ or src/infrastructure/ directly.
- (decision) AD-3: PostgreSQL via Supabase managed Postgres + Prisma ORM. Two connection strings required: DATABASE_URL (direct, for Prisma migrations) and DATABASE_URL_POOLED (PgBouncer, for runtime). Rule: all runtime DB access uses the pooled URL to prevent connection exhaustion on Vercel serverless.
- (decision) AD-4: Auth via NextAuth v5 (Auth.js) — email/password + Google OAuth. Three account types: Parent, Teacher, Admin. Role stored in JWT. Prevents each surface from implementing its own session check. Rule: all route/server-action authorization must read role from the NextAuth session; never trust a client-supplied role claim.
- (decision) AD-5: Child Profile selection is a session-level claim, not a separate auth step. After parent authenticates, selecting a Child Profile stores childProfileId in a signed cookie. Prevents building a separate auth flow for children. Rule: student surface reads childProfileId from the cookie; parent surface reads parentAccountId from the NextAuth session.
- (decision) AD-6: Teacher account approval gate. Teacher accounts carry status: PENDING | APPROVED | REJECTED. NextAuth sign-in callback rejects sign-in when status != APPROVED. Prevents unapproved teachers from accessing the Teacher Portal. Rule: Teacher Portal routes must also verify status === APPROVED server-side; JWT alone is not sufficient.
- (decision) AD-7: Deployment on Vercel (Singapore region) + Supabase Postgres (Singapore region). Prevents latency mismatch between compute and DB. Rule: both Vercel project and Supabase project must be pinned to ap-southeast-1 (Singapore).
- (decision) AD-8: Teacher portal class reports use load-on-visit (Server Components re-fetch on navigation). No WebSockets or SSE in v1. Prevents over-engineering a real-time layer for a next-morning check-in use case. Rule: no persistent connections for teacher report data in v1; deferred to v2.
- (decision) AD-9: Payment via PayOS (automated bank transfer / VietQR webhook). PayOS webhook hits a Next.js route handler at /api/payments/payos/webhook, verifies HMAC signature, activates subscription in DB. Prevents manual admin activation. Rule: subscription status must only change via the verified webhook handler, never via a direct client call.
- (decision) AD-10: Admin panel as custom Next.js route group /admin/* gated by admin role. Covers: Teacher account approval, global session parameter config (question count, free-tier daily allotment), question CRUD. Prevents a separate admin deployment. Rule: every /admin/* route must verify role === admin via NextAuth session server-side.
- (decision) AD-11: Adaptive difficulty as pure domain use case in src/domain/use-cases/adaptive-difficulty.ts. Algorithm: per-Skill sliding window over last N answered Questions (default N=10); Difficulty Level increases when per-Skill accuracy > 80% over the window, decreases when < 50%; N and thresholds are configurable constants. Prevents algorithm logic from leaking into DB queries or UI. Rule: use case takes accuracy history as input, returns next DifficultyLevel; has no imports from @prisma/client or Next.js.
- (decision) AD-12: Question content via seed scripts + admin CRUD. Initial corpus seeded via prisma/seed.ts from structured JSON fixtures. Ongoing authoring via /admin/questions CRUD UI. Prevents questions being hard-coded in application logic. Rule: questions are always DB records; seed fixtures are the source of truth for the initial corpus.
- (decision) AD-13: Question image assets stored in Cloudflare R2, served via public CDN URL. Images referenced in Question records as a CDN URL string. Prevents large binary assets in Supabase Postgres or the monorepo. Rule: Question.imageUrl stores a fully-qualified CDN URL or null; upload flow goes through admin panel -> R2 SDK -> store URL in DB.
- (decision) AD-14: Transactional email via Resend + React Email templates. Triggers: subscription confirmation (parent), teacher account approved/rejected (teacher). Prevents ad-hoc email implementations across surfaces. Rule: all outbound email goes through a single src/infrastructure/email/ adapter; no surface may call Resend directly.
